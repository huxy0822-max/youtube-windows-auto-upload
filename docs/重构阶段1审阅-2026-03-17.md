# 重构阶段 1 审阅说明

这次不是大拆大删，而是先把“本次任务到底要做什么”抽成了一层单独服务，先把主流程理顺。

## 这次实际改了什么

### 1. 新增一层统一任务服务

新文件：

- `run_plan_service.py`

这层现在专门负责 4 件事：

1. 把控制台当前勾选的模块、窗口任务、默认上传规则，整理成一个统一的“本次任务计划”
2. 统一生成预览文本
3. 统一做路径和成品校验
4. 统一决定执行顺序

以后控制台不再自己分别去拼：

- 预览逻辑
- 校验逻辑
- 模拟逻辑
- 正式执行逻辑

而是都先经过这一层。

### 2. 控制台最后生效的入口已经切过去了

`dashboard_app.py` 里这几个最后生效的方法，已经改成调用新服务层：

- `_preview_plan`
- `_validate_paths`
- `_start_simulation`
- `_start_real_flow`

也就是说，现在：

- 预览看到的内容
- 点“路径检查”跑的内容
- 点“模拟 1-2 分钟”跑的内容
- 点“开始真实流程”跑的内容

已经不再是四套互相独立的旧逻辑了。

### 3. 上传阶段也开始吃统一计划

`_run_upload_command` 现在不再重新从界面拼一遍上传计划，而是直接使用这次统一任务服务生成的窗口计划。

这样做的目的就是减少这类老问题：

- 你前面选的是 A，后面上传又自己猜成 B
- 你前面选好了目录，后面又按旧路径规则去找
- 预览说的是一套，实际执行是另一套

## 这次没有硬删什么

`workflow_core.py` 和 `dashboard_app.py` 里原来那些旧函数、重复定义，这次大部分还在。

原因很简单：

- 这次先把主入口切到新层
- 保证现在还能跑
- 下一阶段再逐步删旧逻辑

这样风险比一上来全删低很多。

## 这次完成后的实际效果

现在“本次任务”的主链变成了：

1. 控制台收集你当前勾选的模块、窗口、日期、默认上传规则
2. 新服务层统一生成 RunPlan
3. 预览 / 校验 / 模拟 / 正式执行都用同一个 RunPlan
4. 如果要上传，就把这个 RunPlan 转成窗口计划文件给上传脚本

一句话说：

现在开始把“前面怎么选，后面就怎么执行”这件事收拢了，不再让每一步都自己重新猜。

## 这次已经做过的验证

- `run_plan_service.py` 编译通过
- `dashboard_app.py` 编译通过
- `workflow_core.py` 编译通过
- `batch_upload.py` 编译通过
- `import dashboard_app` 通过
- `import dashboard` 通过
- 新服务层已验证：
  - 能生成统一预览
  - 能统一做路径校验
  - 只上传模式能正常给出可上传目录

## 下一阶段准备继续清什么

下一阶段会继续做这几件事：

1. 把 `dashboard_app.py` 里前面那些旧的重复方法真正清掉
2. 把 `workflow_core.py` 里重复的 `validate_*` 和 `execute_direct_media_workflow` 清成单实现
3. 把“文案生成 / 剪辑 / 上传”再往服务层拆，减少互相串逻辑
4. 把上传阶段里还在做二次猜路径、二次猜素材的兼容逻辑继续收口

## 你现在重点审什么

你只需要看 3 个判断：

1. 以后“预览 / 校验 / 模拟 / 正式执行”都走一套任务计划，这个方向你是否认可
2. 以后上传不再自己重新猜目录，而是优先吃本次任务计划，这个方向你是否认可
3. 下一阶段先删重复定义、再拆服务，而不是直接把全项目推倒重来，你是否认可

如果这 3 个方向你认可，我就继续做阶段 2。
