# 其他设备同步说明（最近改动汇总）

这份文档是给其他设备上的 Codex 看的。

目标只有一个：

- 让其他设备直接同步 `windows-huxy` 分支最近这批改动
- 不用再翻聊天记录
- 不要把旧逻辑又改回去

## 1. 先同步到哪里

统一以这个分支为准：

- GitHub 仓库：[youtube-windows-auto-upload](https://github.com/huxy0822-max/youtube-windows-auto-upload)
- 分支：`windows-huxy`

如果需要稳定 API 基线，可以先参考这个标签：

- `windows-huxy-api-stable-v1.0`

但当前要同步的是 **分支最新代码**，不是只停在稳定标签。

## 2. 这批改动的核心方向

最近这批改动，主要有 6 块：

1. 文案生成改成 API 主链
2. 兜底封面重做并接入模板池
3. `MEGA BASS` 高级视觉预设重做
4. 上传支持每窗口数量和分轮执行
5. 上传页字段简化，但保留窗口级可见性
6. 测试垃圾默认清理

## 3. 文案生成逻辑

### 当前规则

- 标题 / 简介 / 标签：必须走 API
- 不允许本地拼标题
- 不允许旧 `daily/generation_map` 当正式文案来源
- 缩略图：优先图片 API
- 图片 API 连续失败两次，才允许走兜底封面

### Prompt 结构

现在文案请求已经改成：

- **纯自然语言 prompt**
- 不再把任务输入包成一坨 JSON 让模型自己读

并且：

- `masterPrompt` 必须参与
- `titleLibrary` 必须参与
- `musicGenre / audience / angle` 必须原样参与

### 请求方式

- 改成一次 `metadata_bundle` 请求，直接一次返回：
  - 标题
  - 简介
  - 标签
  - 缩略图提示词

### 当前经验

在 `right.codes` 这条链上：

- `gpt-5.4-xhigh` 对完整 metadata bundle 不稳定
- 用户自己实测 `gpt-5.4-high` 能稳定出结果

所以其他设备如果遇到同样问题，优先检查：

- 当前 UI 里选的模型是不是 `gpt-5.4-high`
- `baseUrl` 是否仍是 `https://right.codes/codex`

不要把 prompt 再改回 JSON 结构。

## 4. 兜底封面逻辑

### 当前规则

- 先走图片 API
- 失败 2 次后，改走兜底封面

### 兜底封面新样式

现在不是旧的“黑条 + 随便打字”了，而是：

- 用视频底图/帧图做背景
- 中间做版式化文字覆盖
- 使用筛选后的模板池随机

### 当前允许随机的兜底模板

只允许这 6 套：

- `1`
- `5`
- `7`
- `8`
- `9`
- `10`

不要再把旧的差模板加回去。

## 5. `MEGA BASS` 视觉预设

### 当前目标

不是普通粒子风格，而是更偏：

- 夜店墙
- 中心喷发粒子
- 跟音频轻微联动
- 更粗更硬的字

### 当前已经改好的部分

- 加了 `套用 MEGA BASS 预设` 按钮
- 粒子池扩到 `MEGA BASS` 专用池
- 频谱样式改成 `MEGA BASS` 自己的样式池
- 配色改成 `MEGA BASS` 自己的配色池
- 字体改成 `MEGA BASS` 专用字体池
- 每个视频单独随机，不再整批共用一套
- 删除了底下那条固定不动的假频谱线

### 当前字体池

`MEGA BASS` 相关字体至少包括：

- `Anton`
- `Bebas Neue`
- `Russo One`
- `Orbitron`
- `Teko`
- `Agency Bold`
- `Impact`
- `DIN Bold`
- `Bungee`
- `Black Ops One`
- `Audiowide`
- `Monoton`

### 当前随机规则

只要点了 `套用 MEGA BASS 预设`，随机就只在它自己的库里发生：

- 不要再随机回普通贴纸池
- 不要再随机回普通字体池
- 不要再随机回普通频谱库

如果别的设备上还会随机到旧库，就是没同步到最新逻辑。

## 6. 上传页：每窗口数量 + 分轮执行

### 当前规则

每个窗口现在支持设置：

- `数量(每窗口)`

含义是：

- 填 `1`：这个窗口做 1 套
- 填 `2`：这个窗口做 2 套
- 填 `3`：这个窗口做 3 套

### 执行方式

不是一个窗口连续干完全部数量，而是按轮次：

- 第 1 轮：每个窗口先做 1 条
- 第 2 轮：只做数量 `>=2` 的窗口
- 第 3 轮：只做数量 `>=3` 的窗口

以此类推。

### 每轮规则

- 同一轮内可以并发上传
- 下一轮必须等上一轮结束才开始

不要把这条逻辑改回“一个窗口先连传 3 个再轮到下一个”。

## 7. 上传页字段最近的取舍

### 保留的

“加入窗口时” 这块现在保留：

- `YPP`
- `数量(每窗口)`
- `可见性`
- `分类`
- `儿童内容`
- `AI 内容`
- `窗口定时覆盖`

### 删掉的

- `自定义标题`

原因：

- 标题必须走 API
- 不再保留手填标题入口

### 可见性说明

窗口级 `可见性` 不能删。

现在规则是：

- 上面这排的 `可见性` 是窗口级覆盖
- 下面“统一默认规则”的 `可见性` 是整批默认值

如果窗口没单独改，就走下面默认值。

## 8. 运行结果总结

结束时不能只写一句“全部完成”。

现在要求保留：

- 哪些成功
- 哪些失败
- 哪一步失败

阶段至少区分：

- `Render`
- `Metadata/Cover`
- `Upload`

## 9. 测试垃圾处理规则

最近已经按这个规则收了：

- 所有测试临时目录
- `yt_*`
- `_tmp*`
- `debug`
- `review/grid/layout/smoke`

这些测试垃圾默认用完就删。

其他设备同步时，也要保持这个规则：

- 只要是测试临时文件
- 默认删
- 不要留在项目目录里


## 12. 最后一句话

如果其他设备同步后出现下面这些现象，就说明没同步完整：

- 文案还是走本地
- `masterPrompt` / `titleLibrary` 没生效
- `MEGA BASS` 还是随机到旧贴纸库
- 一个窗口不能设置数量
- 批量上传不是按轮次执行
- `可见性` 只能吃默认值
- 测试垃圾越堆越多
