# 项目功能总纲

这份文档不是代码说明书，而是把你前面提过的要求，整理成一份“你真正想要这个项目最后变成什么样”的总纲。

你审这份的时候，重点看两列：

- `目标规则`：你真正要的最终效果
- `当前状态`：现在代码大概做到哪一步

状态说明：

- `已对齐`：这条方向基本已经做进去了
- `部分对齐`：已经有雏形，但还不够稳，或者批量时还会出问题
- `待继续重构`：这条现在还没彻底落稳，后面要继续改

## 1. 项目最终目标

这个项目最后要做的是一套 **YouTube 音乐频道自动化工作台**。

它不是单纯的上传脚本，也不是单纯的剪辑脚本，而是 3 块能力组合：

1. `生成标题 / 简介 / 标签 / 缩略图`
2. `剪辑`
3. `上传`

这 3 块必须满足：

- 可以单独运行
- 可以两两组合运行
- 可以三块全一起运行
- 同一套任务里支持单窗口，也支持多窗口批量

当前状态：`部分对齐`

## 2. 任务运行方式

### 你要的目标规则

快捷开始页里，应该只需要勾选今天要跑哪些模块：

- `只跑文案`
- `只跑剪辑`
- `只跑上传`
- `文案 + 剪辑`
- `剪辑 + 上传`
- `文案 + 上传`
- `文案 + 剪辑 + 上传`

程序不应该再自己理解成别的模式。

一句话说：

`勾选什么，就只跑什么。`

### 当前状态

- 快捷开始里已经是可勾选模块了
- 但内部主链刚进入“统一任务计划”重构阶段
- 后续还要继续把旧模式残留清掉

当前状态：`部分对齐`

## 3. 批量任务的最小单位

### 你要的目标规则

批量操作的最小单位应该是 `窗口任务`，不是 `tag`，也不是“某个文件夹名”。

也就是：

- 你今天选了哪些 BitBrowser 窗口
- 每个窗口对应什么上传规则
- 每个窗口最终用哪个视频
- 每个窗口最终用哪套标题简介标签缩略图

都应该按 `窗口` 来记。

### 当前状态

- 控制台已经是按窗口加入任务区
- 上传计划也已经有窗口计划文件
- 但执行链里还有一些旧逻辑会按 tag 再次分组处理

当前状态：`部分对齐`

## 4. 路径规则

### 你要的目标规则

以后路径必须清楚，不再乱猜。

应该拆成 4 类路径：

1. `文案输出目录`
   - 存标题、简介、标签、缩略图、manifest、generation_map
2. `音乐目录`
   - 剪辑时读取音频
3. `底图目录`
   - 剪辑时读取图片
4. `成品视频输出目录`
   - 剪辑后视频放这里，上传也从这里拿

另外这些规则必须成立：

- 上传模块不再单独再要一套目录
- 只上传时，直接吃上面这些目录里的现成产物
- 如果目录里没东西，直接明确报错
- 文件夹名字不需要强绑定分组名
- 可以有默认分组绑定目录
- 但实际运行时，以你当前选择/覆盖的目录为准

路径优先级你要的是：

1. `当前任务里手动指定的目录`
2. `该分组长期绑定的目录`
3. `全局默认目录`

### 当前状态

- 路径页面已经基本拆开了
- 分组绑定目录已有雏形
- 当前手动目录覆盖也已有雏形
- 但“上传阶段不再二次猜目录”还在继续重构

当前状态：`部分对齐`

## 6. 素材使用方式

### 你要的目标规则

你希望后面最常用的是这种模式：

- 一个文件夹里直接扔图片和音频
- 不需要改成老格式命名
- 不需要额外再建复杂子目录

然后程序按你今天选的窗口数量来处理：

- 1 个窗口：就取 1 张图 + 1 个音频
- 5 个窗口：就取 5 张图 + 5 个音频
- 如果目录里有 100 张图和 100 个音频，但只选了 5 个窗口
  - 就只生成 5 个视频
  - 剩下的先不动

还要满足：

- 图片和音频不要求一一命名对应
- 已用过的图片和音频要移到 `已用素材目录`，或者删掉
- 上传过的本地视频，按保留天数自动清理

### 当前状态

- 直出模式已有
- 已用素材和保留天数逻辑已有
- 但批量情况下仍然容易和旧目录逻辑冲突

当前状态：`部分对齐`

## 7. 文案生成

### 你要的目标规则

文案系统要支持两套来源：

1. `提示词那套`
   - 走 API
   - 默认优先用这一套
2. `原先那套`
   - 读本地 generation_map / daily content

同时必须支持：

- 多套 API 模板
- 多套内容模板
- 按分组绑定不同模板
- 临时修改当前表单内容后，当前任务就按你现在表单里的内容走
- 不应该偷偷回退到之前测试过的旧模板

另外你要的生成结果要求：

- 每个视频都要有自己独立的标题 / 简介 / 标签 / 缩略图
- 不要重复用上一条视频的文案
- 已用过的文案最好直接删掉，或者至少进历史避免重复
- 已传过的缩略图也不要反复重用

### 当前状态

- 提示词 API 和原先那套都有
- 多模板和分组绑定已有
- 但“批量时每个窗口都保证唯一文案”还需要继续压实
- “旧模板残留 / 旧 manifest 残留”这类问题已经开始修，但还没有彻底宣告完成

当前状态：`部分对齐`

## 9. 剪辑功能

### 你要的目标规则

剪辑这块最后要支持：

- 16:9 成品视频
- 音频多长，视频就多长
- 可以做短时模拟测试
- 真正跑全片时要有进度日志，不然你会以为卡死
- 支持 GPU 编码
- 你的机器是 `AMD CPU + NVIDIA GPU`
- 所以编码重点应该走 `NVENC`

### 当前状态

- 进度日志已有
- NVIDIA 编码已接上
- 当前速度比之前强，但还没到你理想的“极快”
- 继续优化空间主要在特效滤镜链，不是单纯编码器

当前状态：`部分对齐`

## 10. 视觉特效

### 你要的目标规则

视觉特效要支持两种方式：

1. `随机视觉特效`
   - 尽可能降低同质化
2. `手动视觉控制`
   - 你有时候会自己选样式

手动控制里要能配：

- 频谱
- 时间轴
- 黑边
- 缩放
- 颜色
- 粒子
- 贴纸
- 叠字

你还要求：

- 特效池尽量大
- 尤其是雪花、金粉、飘落类
- 位置随机
- 缩放随机
- 漂移随机
- 最好一看就不容易撞模板

### 当前状态

- 高级视觉页已恢复
- 随机池已经扩到 `100+`
- 位置和缩放随机已有
- 但“多层叠加随机”“更丰富的新滤镜类型”还可以继续扩

当前状态：`部分对齐`

## 11. 上传功能

### 你要的目标规则

上传页最后应该满足：

- 单个窗口上传和批量上传合并成一个入口
- 任务区里有几个窗口，就上传几个
- 上传时直接吃当前已经生成好的视频和文案
- 不要中途重新猜别的目录

上传细节你要求固定支持：

- `Altered content = Yes`
- `Category = Music`
- 可设置 `是否儿童`
- 可设置 `可见性`
- 可设置 `定时发布`
- 时区先固定 `Asia/Taipei (+08:00)`
- 上传完成后可选是否自动关闭浏览器窗口

并且：

- 关闭窗口必须是在上传真正完成后
- 不是填完表单就关

### 当前状态

- AI 内容 Yes 已基本打通
- Music 分类已做多轮修复
- 定时发布与时区已有
- 自动关闭开关已有
- 但上传脚本整体仍是当前最脆弱的一块，后续还要继续专项加固

当前状态：`部分对齐`

## 13. 只上传模式

### 你要的目标规则

只上传的时候，不应该还要求目录里必须有图片和音频。

它应该只检查：

- 现成视频在不在
- 现成 manifest / 文案 / 缩略图在不在

如果这些齐了，就应该直接上传。

## 14. 批量去重要求

### 你要的目标规则

一旦今天同时上传多个视频，必须尽量避免：

- 标题一样
- 简介一样
- 标签高度重复
- 缩略图重复

特别是：

- 同一批任务里的多个窗口，不能吃同一条文案
- 之前失败过的旧任务残留，也不能偷偷复用到今天

### 当前状态

- 已经开始做最近历史避重
- 但批量时的唯一性约束，仍然要继续加强

当前状态：`待继续重构`

## 16. 你现在真正最在意的几条

如果只抓最核心，你真正要的是这几件事：

1. 不要再出现“前面选的是这个，后面执行成另一个”的问题
2. 目录不要再乱猜，以当前任务选择为准
3. 单个任务能跑，批量任务也要稳
4. 多窗口时标题简介缩略图不要串台、不要重复
5. 上传页要稳，分类和 AI 声明这些关键项必须打上
6. 随机视觉要足够丰富，尽量降低同质化风险
