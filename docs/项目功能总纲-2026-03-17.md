# 项目功能总纲

这份文档只回答一件事：

`这个项目最后到底要做成什么样。`

我已经按你刚最新确认过的要求，重新整理了一版。

---

## 1. 项目一句话目标

这个项目最后要做成一套 **YouTube 音乐频道自动化控制台**。

它要把下面 3 块能力统一起来：

1. `生成标题 / 简介 / 标签 / 缩略图`
2. `剪辑`
3. `上传`

并且这 3 块必须：

- 能单独运行
- 能任意组合运行
- 能单窗口跑
- 能多窗口批量跑

当前状态：`部分对齐`

---

## 2. 快捷开始最终应该怎么工作

### 目标规则

快捷开始页只负责勾选今天要跑哪些模块。

只需要这 3 个勾选项：

- `生成标题/简介/标签/缩略图`
- `剪辑`
- `上传`

规则很简单：

- 只勾 1 个，就只跑这 1 个
- 勾 2 个，就按顺序连续跑这 2 个
- 3 个都勾，就全流程连跑

程序不应该再额外自己脑补别的模式。

### 现在的理解

这个方向已经开始按新计划层重构，但旧逻辑还没清干净。

当前状态：`部分对齐`

## 3. 文案系统

### 目标规则

文案系统现在只保留 **提示词那套**。

也就是：

- 只走 API
- 只走提示词模板
- 只走 API 模板 + 内容模板这套逻辑

`原先那套 / daily content / generation_map 直接生成文案` 这条老路，后面应该彻底弃用，不再作为正式文案来源。

generation_map 以后如果还保留，也只作为：

- 结果落地文件
- 上传前中间产物
- 历史记录

而不是正式生成逻辑来源。

### 必须支持

- 多套 API 模板
- 多套内容模板
- 模板可保存
- 模板可按 BitBrowser 分组绑定
- 当前任务运行时，以你当前表单保存/绑定后的模板为准
- 不能偷偷回退到旧模板

### 你最重视的硬要求

`批量时每个窗口都保证唯一文案，并且永远不会重复`

这条是重中之重，后面必须做成硬规则，不是尽量。

也就是说：

- 同一批任务里的多个窗口，标题不能重复
- 同一批任务里的多个窗口，简介不能重复
- 同一批任务里的多个窗口，标签组合不能重复
- 同一批任务里的多个窗口，缩略图提示词不能重复
- 不同批任务里的多个窗口，标题、简介、标签、缩略图提示词也不能重复
- 以前已经用过的标题、简介、标签、缩略图提示词，也不能再复用

后面应该上成正式机制：

1. 生成前查全局历史库
2. 生成时传入避重列表
3. 如果撞车，自动重试
4. 生成成功后立刻记入“永久已用历史”
5. 后续任何任务都不再允许复用

### 当前状态

- 提示词那套已经是主路
- 多模板和分组绑定已有
- 避重已经有雏形
- 但“永不重复”现在还没有做到你要求的硬级别

当前状态：`待继续重构`

## 4. 剪辑系统

### 目标规则

剪辑系统负责：

- 从音乐目录读音频
- 从底图目录读图片
- 生成 16:9 成品视频
- 音频多长，视频就多长
- 支持短时模拟测试
- 支持正式全长输出

### 路径上必须清楚

剪辑这块只认这几个路径：

- `音乐目录`
- `底图目录`
- `成品视频输出目录`

不要再让剪辑中途自己去乱猜其他目录。

### 性能要求

你的机器是：

- `AMD CPU`
- `NVIDIA 显卡`

所以剪辑加速的重点应该是：

- 编码走 `NVENC`
- 尽量减少 CPU 成为瓶颈
- 日志里持续显示进度

### 当前状态

- 已有模拟与正式输出
- 已有进度日志
- 已有 NVIDIA 编码
- 但渲染速度和特效链还需要继续优化

当前状态：`部分对齐`

## 5. 视觉特效系统

### 目标规则

视觉特效要同时支持两种模式：

1. `随机视觉特效`
2. `手动视觉控制`

### 手动视觉控制必须保留

你有时候会手动选：

- 频谱
- 时间轴
- 黑边
- 缩放
- 色调
- 粒子 / 贴纸
- 叠字

所以高级视觉页必须长期保留，不能只剩一个随机开关。

### 随机特效的目标

随机特效池要尽量大，重点是：

- 雪花
- 金粉
- 飘落粒子
- 氛围类叠层

并且要做随机：

- 素材随机
- 位置随机
- 缩放随机
- 漂移随机
- 尽量降低同质化

### 当前状态

- 高级视觉页已经接回
- 随机池已经扩到 100+
- 随机位置和缩放已有
- 但还可以继续做多层叠加、更多滤镜类型

当前状态：`部分对齐`

## 6. 上传系统

### 目标规则

上传页是唯一上传入口。

不再区分：

- 单个上传页面
- 批量上传页面

规则应该非常直：

- 任务区里 1 个窗口，就是单个上传
- 任务区里多个窗口，就是批量上传

### 你刚新增的关键硬要求

`批量上传必须并发`

也就是：

- 不能上传第一个，再等第一个完全传完，才开始第二个
- 而是有多少个任务，就同时打开多少个窗口开始上传
- 每个窗口独立走自己的 YouTube Studio 上传流程

一句话说：

`多窗口任务 = 多窗口并发上传`

### 上传细节必须支持

- `Altered content = Yes`
- `Category = Music`
- 可设置 `是否儿童`
- 可设置 `可见性`
- 可设置 `定时发布`
- 时区先固定 `Asia/Taipei (+08:00)`
- 可设置 `上传完成后是否自动关闭浏览器窗口`

### 自动关闭规则

你已经明确过：

- 必须是上传真正完成后才允许关
- 不能只是填完表单就关

### 当前状态

- 上传页已经合并成一个入口
- AI 内容、定时、时区、自动关闭都有基础
- 但“批量并发上传”现在还没有按你最终要求完全做稳
- 上传脚本本身仍是最脆弱的一块，需要专项继续加固

当前状态：`待继续重构`

## 7. 运行中的控制能力

### 目标规则

任务进行中的时候，你希望控制台能做两件事：

1. `暂停`
2. `取消`

用途很明确：

- 出错了可以马上停
- 临时有别的任务占用 GPU / CPU，可以先暂停

### 这块后面建议怎么做

至少要分清楚 2 类暂停：

1. `暂停队列`
   - 不再启动下一个任务
2. `暂停当前执行`
   - 渲染时可以暂停当前 ffmpeg 子进程
   - 上传时可以暂停继续推进下一步点击

取消也要分清楚：

- `取消当前窗口`
- `取消整个批次`

### 当前状态

- 现在主要只有运行状态展示
- 暂停 / 取消还没正式做进主链

当前状态：`待继续重构`

## 8. 路径系统

### 目标规则

路径要彻底解构清楚，至少分成这 4 类：

1. `文案输出目录`
   - 标题
   - 简介
   - 标签
   - 缩略图
   - manifest
   - generation_map

2. `音乐目录`
   - 剪辑读取音频

3. `底图目录`
   - 剪辑读取图片

4. `成品视频输出目录`
   - 剪辑输出视频
   - 上传直接从这里拿视频

另外继续保留：

- `FFmpeg`
- `已用素材目录`
- `上传后保留天数`

### 上传路径规则

上传不应该再单独有一套目录逻辑。

它直接吃上面已经配置好的目录：

- 只上传时
  - 直接检查成品视频输出目录和文案输出目录里有没有现成结果
- 剪辑并上传时
  - 直接吃本次刚产出的目录结果

如果目录里没有东西，就直接明确报错。

### 目录优先级

以后必须按这个顺序：

1. `当前任务手动覆盖的目录`
2. `当前分组长期绑定的目录`
3. `全局默认目录`

### 你明确过的硬要求

`文件夹名字不要再跟分组强绑定`

可以有默认绑定，但最终必须以你当前选的目录为准，而不是死按分组名去拼目录。

### 当前状态

- 已经开始往这个方向改
- 但上传阶段和旧 fallback 逻辑还没完全清掉

当前状态：`部分对齐`

实际运行时：

- 你当前任务里如果手动改了目录
- 就必须以当前手动目录为准

- 但整个项目里所有入口还没有完全统一按这条规则执行

## 10. 素材使用方式

### 目标规则

你最常用的应该是这种直出模式：

- 一个目录里直接放图片
- 一个目录里直接放音频
- 或者按你当前绑定目录来读
- 不强迫改成老命名格式

程序按今天选中的窗口数来取素材：

- 选 1 个窗口，就处理 1 套
- 选 5 个窗口，就处理 5 套
- 就算目录里有 100 张图和 100 首歌，如果只选 5 个窗口，就只做 5 个

### 已用素材处理规则

你前面已经定过：

- 已用过的图片和音频，移到 `已用素材目录`，或者直接删除
- 已用过的文案，也不要再次参与生成
- 已传完的视频，按保留天数自动清理

### 当前状态

- 素材直出模式已有
- 已用素材目录已有
- 但批量时和目录优先级相关的逻辑仍需继续统一

当前状态：`部分对齐`

3. 所有生成结果落永久历史索引
4. 新生成前先查历史索引
5. 冲突就重试，不允许直接复用
