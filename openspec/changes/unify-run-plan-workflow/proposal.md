# Proposal: unify-run-plan-workflow

## 背景

当前仓库已经能跑通单次流程，但主链路仍然存在结构性问题：

- 文案生成、剪辑、上传三块没有真正解耦。
- 上传阶段仍然会重新按 tag、目录、manifest 去猜输入。
- UI、运行状态、长期配置、本次运行产物混在一起。
- 批量任务以 tag 为中心，而不是以窗口任务为中心。
- 分组列表、窗口计划、channel mapping 之间缺少明确优先级。

这些问题会导致：

- 单窗口可用，多窗口容易漂移。
- 同一批次多视频时，标题/简介/缩略图容易重复或错配。
- 路径覆盖、分组绑定、旧 manifest 之间互相打架。
- 每次修 bug 都倾向于继续在 `batch_upload.py` 和 `dashboard_app.py` 里打补丁。

## 目标

本次重构要把系统改成一个明确的三段式流水线：

1. `Metadata`: 只生成标题/简介/标签/缩略图
2. `Render`: 只根据素材和视觉配置生成视频
3. `Upload`: 只根据显式产物执行上传

这三段都必须：

- 能独立运行
- 能任意组合运行
- 在批量场景下保持同一套规则

## 关键原则

### 1. 单一事实来源

本次运行的所有输入、路径、窗口、上传默认项，都必须先汇总成 `RunPlan`。

后续模块只消费 `RunPlan`，不再自己推断。

### 2. 任务最小单位是窗口任务

以后系统的执行单位不再是“tag 目录”，而是：

- 一个窗口
- 一组覆盖规则
- 一组已分配素材
- 一组独立产物

### 3. 路径规则固定

目录解析顺序固定为：

1. 当前任务覆盖目录
2. 当前分组绑定目录
3. 全局默认目录

不再允许上传阶段再按 tag 名去拼目录。

### 4. 上传阶段只消费显式产物

上传时必须已经拿到：

- `video_path`
- `title`
- `description`
- `tag_list`
- `thumbnail_path`
- `container_code`
- `upload_options`

上传阶段不再去“猜”这些值。

## 非目标

这次重构不解决这些问题：

- 更换 UI 技术栈
- 换成网页后台
- 新增新的模型供应商
- 重做视觉特效算法
- 重做 BitBrowser API 适配层

这些都可以在主链稳定后再继续做。

## 风险

### 1. 现有兼容链很多

目前仓库里存在：

- 老 manifest 逻辑
- `generation_map.json`
- 分组绑定目录
- 旧目录格式
- 旧 tag_to_project 配置

如果一次性全部砍掉，短期会影响现有可运行链。

### 2. UI 依赖较重

`dashboard_app.py` 当前既承担 UI，也承担很多业务拼装逻辑。重构时如果先动 UI，会导致测试面过大。

## 建议实施策略

先建新主链，再逐步切换 UI。

顺序为：

1. 先定 `RunPlan` 和新 artifact 目录结构
2. 再抽出 Metadata / Render / Upload 三个服务
3. 最后让 UI 接到新服务上

## 审阅点

这份 proposal 需要先确认 4 件事：

1. 是否同意以后系统只认 `RunPlan`
2. 是否同意以后任务最小单位是窗口任务
3. 是否同意路径优先级固定为“任务覆盖 > 分组绑定 > 全局默认”
4. 是否同意按“先新建服务层，再切 UI”的顺序实施
